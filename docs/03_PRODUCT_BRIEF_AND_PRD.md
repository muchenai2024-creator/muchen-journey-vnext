# 03｜产品简报与 PRD

状态：`APPROVED_FOR_BUILD`  
版本：V0.1  
日期：2026-07-20  
文档 Owner：Product Owner  
批准结论：首版只做探索营闭环；`DEC-004` 已批准。

## 1. 一句话定义

Muchen Journey vNext 是一个把新人当前任务、真实提交、主管反馈和下一步组织成连续闭环的成长产品；它帮助新人知道“现在该做什么”，帮助主管知道“现在该评谁、如何给出有效反馈”。

## 2. 要解决的问题

### 学员问题

- 不清楚自己当前处于什么阶段；
- 同一状态在多个页面重复或冲突；
- 任务说明、材料、测验、提交和结果分散；
- 提交后不知道是否成功、谁会反馈、何时能继续；
- 被要求修订时，旧版本与当前版本容易混淆。

### 主管/Reviewer 问题

- 队列、身份、班级、证据和评分入口不在同一工作流；
- 无法快速判断哪一条最需要处理；
- 评审材料不完整时缺少明确阻断；
- AI 建议、人工判断和最终结论边界不清；
- 通知失败与业务失败容易被混为一谈。

### 运营问题

- 邀请、加入、主管分配、内容版本和异常处理缺少统一事实；
- 历史记录和当前执行状态混合；
- 依赖人工查表、补数据或在多个系统间核对。

## 3. P0 产品假设

如果系统只呈现一个权威当前行动，把任务、提交、评审、结果和下一步连接成服务端权威闭环，并让异常状态提供明确恢复动作，那么新人能在较少协助下完成探索营，主管能按时给出可追溯反馈，运营不再依赖临时数据修复。

## 4. 目标用户与角色

| 角色 | 核心目标 | P0 权限边界 |
| --- | --- | --- |
| Learner/新人 | 完成当前任务、收到反馈、进入下一步 | 只访问自己的任务、提交、结果和允许的内容 |
| Reviewer/直属主管 | 处理明确分配给自己的评审 | 只访问授权对象；不能代替学员提交或覆盖历史结论 |
| Operator/运营 | 创建邀请、确认加入、分配主管、处理异常 | 按组织/批次范围操作；所有变更留审计 |
| Admin | 管理配置与紧急修复 | 最小授权；高风险操作要求理由和审计 |
| AI Advisor | 提供可降级的初评建议 | 不是身份角色，不拥有最终判断和状态迁移权限 |

## 5. P0 范围

### REQ-BR-001｜邀请与加入

运营创建有期限、可撤销、用途明确的邀请；新人使用邀请进入登记。邀请不可重复滥用，失败状态有恢复路径。

### REQ-BR-002｜身份与会话

新人通过邀请完成最小身份确认并获得 vNext 独立会话，不强制飞书绑定；非本地 Reviewer/Operator 后续绑定独立飞书身份。业务实体只引用内部 `user_id`。

### REQ-BR-003｜当前行动

登录后只返回一个权威当前行动：待加入、Day 0、当前任务、等待评审、需要修订、已通过或已完成。用户能在 5 秒内说明自己现在该做什么。

### REQ-BR-004｜任务理解与提交

任务页包含目标、完成标准、所需材料、当前草稿/附件和一个主提交动作。提交创建不可变版本；安全重试不产生重复事实。

### REQ-BR-005｜主管评审

主管看到按优先级排序的授权队列，能够开始评审、查看固定提交版本、完成必填评价并提交最终结论。

### REQ-BR-006｜修订闭环

结论为“需要修订”时，学员看到明确反馈和唯一修订动作；新提交产生新版本，旧提交与旧评审只读保留。

### REQ-BR-007｜通过、结果与交接

结论为“通过”时，系统原子记录最终评价、结果和下一步/交接。若异步通知失败，结果仍成立且可恢复通知。

### REQ-BR-008｜运营处理

运营能查看邀请、加入、主管分配和异常状态；P0 不提供通用数据库编辑器。纠错必须使用有意图、有校验、有审计的命令。

### REQ-BR-009｜通知

关键事件可发送飞书通知；通知失败只影响通知状态，不回滚业务事实。收件人与应用身份必须可审计。

### REQ-BR-010｜不可变历史

提交、评审、结果、状态变化和高风险运营命令可按权限追溯；禁止覆盖旧正文或旧结论。

## 6. P0 非目标

- 新手村、AI 学院、Talent OS、职业公会和真实项目交付；
- 多空间 Shell、全站 route registry 和兼容旧路由；
- 完整课程系统、考试认证和证书；
- 积分、排名、商城、虚拟人物和服饰；
- AIGC 视频或规模化内容生成；
- AI 自动通过/拒绝、自动录用、项目准入或人事判断；
- 通用低代码后台、任意表编辑器和复杂报表；
- 全量历史迁移；
- 旧系统实时双读、双写、fallback 或旧页面回退。

## 7. 核心用户故事

### US-001｜新人首次进入

作为受邀新人，我希望在完成最少登记后看到唯一当前步骤，以便不需要说明会就能开始。

验收：邀请有效；身份建立；页面显示当前阶段、当前任务、反馈人/反馈预期和一个主 CTA。

### US-002｜新人安全提交

作为新人，我希望网络抖动或重复点击不会生成两份提交，以便确认我的成果没有丢失或重复。

验收：相同 idempotency key + 相同正文返回同一结果；相同 key + 不同正文被拒绝；草稿保留。

### US-003｜主管开始评审

作为主管，我希望一进入工作台就知道先评谁并能立即开始，以便减少查找和误评。

验收：只显示授权记录；优先原因明确；证据不完整时阻断最终提交；GET 无副作用。

### US-004｜修订再提交

作为被要求修订的新人，我希望看见具体反馈和当前待改版本，以便不会重复旧提交或丢失历史。

验收：旧版本只读；新版本号递增；新评审固定引用新版本。

### US-005｜通过并进入下一步

作为新人，我希望通过后立即看到结果与唯一下一步，以便知道探索营是否完成以及后续由谁接手。

验收：结果与下一步在同一权威响应可读；通知延迟不会让结果消失。

## 8. 体验与业务约束

- 每个页面只回答一个核心问题；可行动状态只有一个主 CTA。
- 服务端状态决定可执行动作；客户端不得推导第二套状态机。
- AI 输出明确标记“建议”，人工结论明确标记责任人。
- 游戏化只用于目标、进度和完成反馈，不改变资格或人事判断。
- 错误必须说明：发生了什么、是否已保存、用户下一步和支持 request ID。
- 不通过颜色单独传达状态；主路径支持键盘操作和 390/768/1280 视口。

## 9. 成功指标

P0 试点在上线前先记录基线，避免使用没有分母的目标值。批准指标：

| ID | 指标 | 口径 | 护栏 |
| --- | --- | --- | --- |
| KPI-001 | 无协助闭环完成率 | 进入后无需人工讲解完成指定闭环的人数/有效参与人数 | 不能通过简化质量标准提高 |
| KPI-002 | 当前行动理解率 | 5 秒测试中正确说明下一步的人数/测试人数 | 分角色、分状态记录 |
| KPI-003 | 首次有效行动时间 | 进入页面至首次有效业务命令的时间 | 排除加载/系统故障 |
| KPI-004 | 评审周转时间 | 有效提交至首次人工结论 | 同时监控评审质量和退回率 |
| KPI-005 | 重复事实率 | 重复提交/结论/通知业务事实数 | 目标为 0 |
| KPI-006 | 支持介入率 | 需要运营手工修复或讲解的参与者占比 | 分类记录根因 |
| KPI-007 | 状态冲突率 | 同一实体出现互相矛盾展示/命令的事件数 | 目标为 0 |

`DEC-010` 已批准 14 天试点阈值：完成率 ≥80%、当前行动理解率 ≥90%、90% 评审在两个工作日内、重复事实/状态冲突为 0、支持介入率 ≤20%。不得用“感觉更顺畅”替代实测结论。
