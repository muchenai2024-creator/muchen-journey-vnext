# 01｜上一轮重构失败深度复盘

状态：`APPROVED_FOR_BUILD`  
版本：V0.1  
日期：2026-07-20  
复盘对象：2026-07-07 至 2026-07-20 的 Muchen Journey 重构、探索营 V2/P0/P1 与生产修复过程  
复盘原则：分析系统和流程，不归咎个人；数据来自本地文档、Git 历史和当前代码结构。

## 1. 执行结论

这次重构不是“做得不够彻底”，而是从一开始就被定义成了在旧系统内部增建新系统。

文档口头上要求“旧系统停止修补”和“新事实源”，但实际执行合同同时要求保留旧能力、复用旧基础设施、在旧仓库旁路建设 V2、保留兼容入口和回退开关。结果是：旧代码没有退场，新代码以 P0、P1、V2 和适配层的形式叠加在同一个应用里；每新增一条路径，就需要同时维护旧行为、兼容行为、新行为以及覆盖三者的门禁。

因此 bug 增长不是偶发质量问题，而是架构和交付模型的可预期结果。

## 2. 事实证据

### 2.1 产品定义从源头否定了 Greenfield

当前仓库 README 明确把 Muchen Journey 定义为旧 Muchen Quest/新手村的升级迭代，要求保留登录、任务、审核、证据、积分、阶段评审和运营逻辑，并写明“不是全新应用”。这与“从零独立的新系统”相反。

页面减重审计把 61 个 canonical UI、5 个别名和 39 个 legacy 路由共同纳入治理，总路由达到 105；它优化了混合系统，却没有缩小系统边界。

探索营 V1 冻结合同明确规定：V1 在 V2 旁边继续可用；V1/V2 共享加入入口；生产数据库、Alembic 历史、身份、审核证据和发布流程继续保留。冻结的是文件哈希，不是运行时依赖。

### 2.2 “新版本”实际是增量叠加

从探索营 V1 冻结提交 `bca20e4` 到 P0 候选 `ec71d35`：

- 125 个文件发生变化；
- 新增 12,578 行，只删除 109 行；
- 新增 `journey_p0`、`exploration_v2`、`/journey` 和 `/workbench`；
- 同时保留原有 `academy`、`ai-academy`、`exploration-camp`、`launch`、`learning`、`village`、旧后台和旧 API；
- 新数据库结构以 `0017_p0_journey_vertical_slice.py` 接在旧 Alembic 历史之后。

从旧生产基线 `90737a9` 到当前 `ef37f9a`：645 个文件变化，新增 82,464 行，删除 8,512 行。这个比例说明主要策略是增加隔离层、合同、路由、脚本和新实现，而不是建立独立产品并移除旧复杂度。

### 2.3 当前代码仍同时承载多代产品

当前前端同时存在：

- `/academy` 与 `/ai-academy`；
- `/exploration-camp` 与 `/exploration-camp/v2`；
- `/journey`、`/dashboard`、`/today`、`/learning`、`/launch/*` 等多套入口；
- `frontend/lib/adapters`、`frontend/lib/mock`、`frontend/lib/p0-api`、V1 本地状态和 V2 服务端状态；
- 旧路由 canonicalize、redirect、feature flag、fallback 和 compatibility 逻辑。

后端同一个 FastAPI 进程同时装配 `exploration_camp`、`exploration_v2`、`journey_p0`、Academy 与 legacy platform 路由。所谓 legacy quarantine 仍然共享进程、配置、数据库、部署和故障域。

当前源代码约 132,957 行。大型热点包括：

- 6,500 行浏览器验收脚本；
- 6,218 行飞书多维表格适配器；
- 2,218 行探索营 V1 单体客户端；
- 1,517 行路由注册表；
- 1,356 行页面合同注册表；
- 1,384 行探索营 V2 后端路由；
- 1,600 行 P0 核心服务。

代码搜索显示约 120 个文件涉及 `legacy`、96 个文件涉及 `v1`、86 个文件涉及 `v2`、55 个文件仍出现 `localStorage`。这些数字不是单独的 bug，但说明兼容语义已经扩散成横切关注点。

### 2.4 交付速度超过了架构收敛速度

当前 Git 元数据包含 872 个提交、257 个本地/远程分支引用和 112 个 worktree。仅 2026-07-15 至 2026-07-20 当前主线就出现 127 个提交，其中 54 个以 `fix` 开头，23 个以 `feat` 开头。

高频改动集中在 Makefile、发布前检查、部署脚本、UAT 账本、浏览器 smoke、P0 CLI 和核心服务。团队大量时间用于让门禁、环境、身份、回滚和兼容路径重新对齐，而不是稳定产品域。

### 2.5 测试数量增长没有阻止体验与集成缺陷

前端 `package.json` 已包含 93 个脚本，约 70 个 acceptance 文件；合并门禁串联几十个静态、契约、身份、旧版隔离、空间、响应式和浏览器检查。

问题不在于“测试太多”，而在于测试拓扑继承了产品拓扑：旧路径、canonical 路径、V2 路径、P0 路径、各空间和不同环境都需要各自证明。测试本身开始承担兼容系统的复杂度，并成为新的高频故障面。

冻结哈希能证明旧文件没有变化，却不能证明新系统没有依赖共享的 auth、route registry、数据库表、构建输出或部署环境。许多门禁证明的是“合同和代码一致”，不是“用户实际只有一个系统”。

## 3. 原计划与实际结果

| 目标 | 文档表达 | 实际实现 | 后果 |
| --- | --- | --- | --- |
| 冻结旧系统 | 旧系统不再修补 | 旧系统继续在同一应用运行，并允许数据隔离、评分、共享入口和可靠性修复 | 边界持续移动 |
| 单一事实源 | PostgreSQL 权威，不长期双写 | 新表接在旧数据库迁移链；旧 JSON、本地状态、飞书投影和适配器继续存在 | 状态解释需要兼容映射 |
| 新垂直切片 | `/journey`、P0 API 与新状态机 | 直接加入旧 repo、旧 shell、旧 auth、旧 route registry 和旧部署 | 新代码受旧回归矩阵约束 |
| V1/V2 隔离 | 新代码放到 `/v2` 前缀 | 两代共用入口、进程、库、配置、发布与回退开关 | 无物理故障隔离 |
| 旧版只读退役 | D1-D4 逐步删除 | 删除条件被推迟到未来，当前仍需继续兼容 | 每次修复都要考虑旧调用者 |
| 质量门禁 | 大量 contract/acceptance/UAT | 门禁数量持续增长且频繁修复门禁本身 | 交付反馈慢，假绿/环境红灯增多 |
| 单一文档事实源 | 8 份文档与 Decision Register | DOCX 位于顶层未提交仓库；根 Git 甚至没有首个 commit；后续生产事实追加回原设计稿 | 文档无法可靠 diff、review、回滚和绑定代码版本 |

## 4. 根因分析

### 根因 A：项目类型定义错误

团队把“替代旧系统”当作“在旧系统中建设下一代模块”。只要新版本仍必须在同一个 repo、进程、数据库和发布链旁路运行，它就不是独立重写，而是渐进式 strangler/兼容性重构。

这类策略本身并非错误，但它与本次期望的“彻底区隔”不一致。错误在于名称和验收标准不一致。

### 根因 B：隔离被定义成代码所有权，而不是物理依赖

上一轮建立了 worktree、owner、受保护路径、冻结哈希、route owner 和 domain boundary，但没有要求：

- 新仓库不能导入旧源码；
- 新运行时不能访问旧 API；
- 新数据库账号不能连接旧数据库；
- 新迁移历史必须从 `0001` 开始；
- 新部署必须在旧应用离线时仍可独立启动；
- 新版回滚只能回滚到上一新版本，而不能回到旧写路径。

于是“隔离”变成组织标签，没有成为系统属性。

### 根因 C：兼容和退役条件被后置

文档把 legacy 删除安排为 D1-D4，将 V1 保留为只读兼容和安全回退，并允许 V1/V2 共享入口。这等于在新功能尚不稳定时，把团队同时置于建设、迁移、兼容、修复和运维五种工作模式。

退役没有明确截止时间、调用量阈值和强制 Owner；兼容路径自然长期存在。

### 根因 D：开发在关键决策关闭前开始

旧 Decision Register 中前端栈、组件库、Token、数据保留、试点批次、发布门槛等仍是开放项，但实现已经进入 P0 候选和生产。后续每关闭一个决策，都要反向修改代码、门禁和部署。

文档更多像“边开发边解释当前做法”，而不是开发前不可越过的输入。

### 根因 E：变更吞吐量没有 WIP 上限

大量短生命周期分支、worktree、修复提交和并行空间包让每个候选的基线快速漂移。即使每个局部变更都合理，集成系统也无法在同一速度下形成稳定心智模型。

缺少以下约束：一个发布候选、一个合同 Owner、共享文件串行修改、稳定观察窗口和 bug 趋势门禁。

### 根因 F：质量体系偏向结构证明，弱于系统简化证明

测试验证了路由、selector、哈希、合同、权限拒绝和自动化输出，但没有把以下问题设置为一票否决：

- 新版本在旧 repo 不存在时能否构建？
- 新数据库与旧表完全不可达时能否运行？
- 用户是否仍可能走到旧页面？
- 一个业务概念是否只有一个模型和一个命名？
- 修一个 bug 是否需要同时改两代实现或三个门禁？

质量门禁确保复杂系统“没有明显断裂”，却没有阻止系统继续变复杂。

## 6. 哪些成果应保留

重做不等于否定上一轮全部产品思考。以下内容可以作为业务需求重新表达，但不得复制实现：

- 学员始终看到一个当前行动和一个明确下一步；
- NPC/主管使用优先队列和独立评审工作台；
- 服务端数据库是唯一业务事实源；
- 提交、评审、证据和审计保留不可变历史；
- 写命令支持幂等、乐观并发和安全重试；
- AI 只能提供可降级建议，人工拥有最终结论；
- 权限必须服务端执行，前端隐藏不是授权；
- 真实角色 UAT、备份恢复和发布观测是上线证据；
- 游戏化反馈不能影响能力认证或人事判断。

保留的是被验证的业务原则，不是旧类名、表名、API、路由、状态枚举、组件、脚本或部署形态。

## 7. 下一版必须改变的机制

1. 项目名必须叫 Greenfield 重建，不再叫止血重构、V2 旁路或兼容升级。
2. 先通过 G0 文档门禁，再初始化代码。
3. 新仓库、新数据库、新迁移、新部署、新环境和新可观测项目一次性建立。
4. 旧系统只允许离线导出；新系统没有旧 API、旧表和旧包的网络权限。
5. 首版只有一个垂直闭环和一套路由，不建设多空间平台。
6. 需求、状态、接口、数据和验收以永久 ID 追溯。
7. 任何 bug 修复必须补回归场景并回答“为什么现有门禁没有发现”。
8. 控制 WIP：一个候选、短分支、共享合同单 Owner、固定观察窗口。
9. 兼容发生在外部切换层或一次性导入层，不进入核心业务代码。
10. 独立性用断网/断库/移除旧仓库的自动化验收证明。
